The **shell suite** is four small packages that work together to put a real PTY on the other end of a browser tab. You pick the transport (WebSocket for the simple case, WebRTC P2P for NAT-traversing peer connections) and the packages take care of the rest.

## How the four packages fit together

```
                      ┌─────────────────────────┐
                      │   @beamhop/shell-protocol   │  ← shared message types
                      └────────────┬────────────┘
                                   │
                ┌──────────────────┼──────────────────┐
                │                                     │
   ┌────────────▼────────────┐         ┌──────────────▼──────────┐
   │  @beamhop/shell-client  │ ←──WS──→│  @beamhop/shell-server  │
   │       (browser)         │ ←──P2P─→│  (host, wraps node-pty) │
   └─────────────────────────┘         └─────────────────────────┘
                                                     ▲
                                                     │ optional, only for P2P
                                                     │
                                       ┌─────────────┴───────────┐
                                       │  @beamhop/shell-relay   │
                                       │  (WebSocket signaling)  │
                                       └─────────────────────────┘
```

`shell-protocol` defines the wire format. `shell-client` and `shell-server` both depend on it. `shell-relay` is a tiny self-hosted signaling endpoint you only need if you're using the P2P transport with the `ws-relay` strategy.

<Aside type="caution" title="Security model">
  Treat access tokens like SSH keys. The WebSocket server defaults to binding `127.0.0.1` for a reason. P2P offers end-to-end encryption via a shared password — see the server reference for the full threat model.
</Aside>
